## 一、设计目标

招聘题第七章的 IM 验收题里，RAG 知识库需要稳定处理三类问题：

1. **带引用命中**：年假、报销材料、入职第一天等制度类问题必须从知识库返回，并附文件、章节/页码/段落和原文 quote。
2. **扩展问法命中**：加班补偿、出差补贴、病假申请、年假余额等口语化问题应有足够样本文档覆盖。
3. **干净拒答**：CEO 家庭住址、2030 年销售目标、员工薪资明细等库中无依据或涉及隐私的问题必须拒答，禁止编造。

核心取舍：样本文档不是越多越好，而是保证“该命中的稳命中、该拒答的干净拒答”。本次增强保持 `data/samples` 目录 8 个文件以内，避免违反单目录文件数红线。

## 二、样本文档矩阵

当前样本文档共 **7 篇**，覆盖 `md/docx/txt/pdf` 四种解析格式：

| 文档 | 格式 | 主要职责 | 重点覆盖 |
|------|------|----------|----------|
| `员工手册.md` | MD | 公司制度权威源 | 年假、打卡补卡、病假/事假、加班补偿、报销总则、出差原则、账号安全 |
| `新人入职指南.md` | MD | 入职流程权威源 | 入职前材料、入职第一天、账号设备、首周培训、试用期与转正 |
| `FAQ.md` | MD | 高频问答兜底 | 年假余额、跨年年假、补卡、报销流程、病假、出差补贴、权限申请 |
| `信息安全与账号权限.md` | MD | 安全与权限制度 | 基础账号、岗位账号、权限申请/回收、设备安全、VPN、资料共享 |
| `休假与福利政策.txt` | TXT | 休假与福利补充 | 年假余额、病假、事假、婚产相关假、福利领取、休假交接 |
| `报销制度.docx` | DOCX | 报销细则 | 发票材料、审批流程、差旅报销、特殊超标、退回补充、不予报销 |
| `差旅与报销标准.pdf` | PDF | 差旅标准 | 交通、住宿、餐饮补贴、市内交通、超标审批、页码引用 |

设计原则：

- Markdown 文档用多级标题保留 `heading_path`，便于引用展示和原文定位。
- TXT 文档用于验证无标题格式的段落定位，依赖 `paragraph_index`。
- DOCX 文档保留 Word 标题样式，验证 `heading_path`。
- PDF 文档作为静态样本提交，验证 `page_number` 引用定位；不新增运行时 PDF 生成依赖。

## 四、检索与拒答边界

| 项目 | 当前策略 |
|------|----------|
| 分块 | `RAG_CHUNK_SIZE=800`，`RAG_CHUNK_OVERLAP=120` |
| 检索 | pgvector 余弦相似度，`RAG_TOP_K=5` |
| 阈值 | `RAG_SCORE_THRESHOLD=0.72`，不因召回不足降低默认阈值 |
| 引用 | 返回 `filename`、`heading_path`、`page_number` 或 `paragraph_index`、`quote`、`score` |
| 拒答 | 检索为空、低于阈值、隐私敏感、未来无依据问题均拒答 |

边界约束：

- 样本文档不写 CEO 住址、个人薪资、2030 销售目标等内容，避免误召回。
- RAG 回答必须带引用；没有可引用依据时采用统一拒答文案。
- 若后续补充真实公司文档，需重新跑验收题，特别检查隐私和未来无依据问题是否仍拒答。

## 五、维护流程

1. 更新样本文档后，通过 Web 后台上传或调用索引流程重新入库；同名文件应触发旧文档和旧 chunk 软删除。
2. 跑解析测试：`cd apps/api && uv run pytest tests/test_parsers.py`。
3. 有数据库时跑评测脚本：`uv run python scripts/eval.py`，重点看 knowledge/refuse 用例。
4. 手动在 IM 或 Web 调试页验证：年假、报销、入职、加班、出差、病假、年假余额命中带引用；CEO 住址和 2030 销售目标拒答。
