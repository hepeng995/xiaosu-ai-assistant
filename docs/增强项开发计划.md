## 一、背景

小苏 AI 助手已完成招聘笔试要求的第 0–10 阶段（共 11 个阶段），代码落地、测试通过、已上 public 仓库（[`hepeng995/xiaosu-ai-assistant`](https://github.com/hepeng995/xiaosu-ai-assistant)）。

本文档汇总 `自评.md` / `AI_USAGE.md` 中自列的 **5 处后续增强项**，给出每项的目标、现状、实现步骤、改动范围、风险与验收方式，作为后续迭代的开发计划。

> **红线不变**（见 `开发规范.md`）：
> - 工具选择禁硬编码（由 LLM 按 Tool Schema 自主决定）
> - RAG 必带引用、无依据必拒答
> - IM 异常必有兜底回复
> - `.env`/密钥不入库、日志禁止明文打印
> - 单文件 ≤ 500 行、单目录 ≤ 8 文件
> - 每阶段 1–3 个规范化 commit（`<type>(<scope>): <subject>`）
> - 增强项须保持「未配置时降级为 mock」原则，不破坏当前无 key 全流程可跑的特性

## 三、逐项详述

### 3.1 增强项 1：真实 LLM 联调

**目标**：配置真实 API Key，启用真实 function calling（工具自主选择）+ 真实语义向量检索，替代当前的确定性 mock。

**现状（重要：代码已就绪）**
- `app/llm/embedding.py::_embed_via_api()` 已实现真实 embedding（调用 OpenAI-compatible `/embeddings`，带重试、`dimensions`）
- `app/llm/openai_compatible.py::_chat_with_tools_api()` 已实现真实 function calling
- 两处 `use_mock` 在 key 为 `replace_me` 时为 True，**配真实 key 即自动切换真实模式**

**实现步骤**
1. 配置 `.env`（用户操作，不入库）：填入真实 `LLM_API_KEY` / `LLM_BASE_URL` / `LLM_MODEL` 与 `EMBEDDING_API_KEY` / `EMBEDDING_BASE_URL` / `EMBEDDING_MODEL`
2. **`EMBEDDING_DIMENSION` 必须与所选 embedding 模型一致**（text-embedding-3-small = 1536 / bge-large-zh = 1024；`89b6c5a` 已支持自定义维度）
3.（可选增强）真实 token 级流式：`routes_chat.py` 当前为伪流式（先全量生成再按字符分片）；可补 `openai_compatible.py::chat_stream()`（httpx SSE 读 `stream=True`）+ `agent.py` 最终回答阶段流式产出 + `routes_chat.py` 直接 yield 真实 token
4. 端到端验收：跑 README 9 道验收题

**改动文件**：`.env`（用户）；`app/llm/openai_compatible.py`、`app/agents/agent.py`、`app/api/routes_chat.py`（可选流式）
**复用现有抽象**：`use_mock` 开关、`core/config.py::is_secret_configured`、现有真实分支
**风险**：真实 key 严禁入库/打印日志；维度与模型不符会导致检索全错
**验收**：9 道题全部符合预期（知识库命中带引用、工具选对、多轮指代、隐私/无依据拒答、无效 key 兜底）

### 3.2 增强项 2：钉钉沙箱真实联调

**目标**：接入真实钉钉机器人，群内 @ 小苏完成核心问答 + 多轮 + 引用 + 兜底。

**现状（代码已就绪）**
- `app/im/dingtalk.py`（`verify_sign` / `parse_webhook` / `reply_via_webhook`）+ `routes_im_dingtalk.py` 完整链路已实现
- 当前回复依赖 payload 里钉钉回传的 `sessionWebhook`（被动回复够用）

**实现步骤**
1. 钉钉开放平台配置（用户操作）：创建企业内部应用 → 加机器人能力 → 取 `AppKey` / `AppSecret` / `RobotCode`；消息接收地址填 `https://<ngrok 域名>/api/im/dingtalk/callback`；测试群部署机器人
2. 填 `.env`：`DINGTALK_APP_KEY` / `DINGTALK_APP_SECRET` / `DINGTALK_ROBOT_CODE` / `DINGTALK_CALLBACK_TOKEN`
3. 公网穿透：ngrok / frp 暴露 `localhost:8000` 为公网 HTTPS
4.（可选）主动发消息：`im/dingtalk.py` 补 `get_access_token()`（AppKey/Secret 换 token）+ 钉钉发消息 API
5. 验证：群内 @ 问「员工 001 哪个部门」，确认真实 function calling + 引用

**改动文件**：`.env`（用户）；`app/im/dingtalk.py`（可选主动发消息）；`.env.example`（补字段说明）
**风险**：生产必须开启验签（当前未配 secret 时放行告警）；回调必须 HTTPS
**验收**：群聊 @ / 私聊均能问答、多轮上下文、引用展示、异常兜底、后台可见 IM 日志

### 3.3 增强项 3：集成测试（RAG 拒答 / 同名替换 / 文档索引）

**目标**：补齐文档要求的 3 个集成测试，连真实 pgvector 验证索引/替换/拒答闭环。

**现状**：无 `conftest.py`，现有 4 个测试均为不依赖 DB 的单元测试。

**实现步骤**
1. 新建 `apps/api/tests/conftest.py`：async DB session fixture，连测试专用库（如 `xiaosu_test`），每测前清表 / 事务回滚保证幂等；提供小 md 文本 fixture（可选 `testcontainers-python` 做进程级隔离）
2. `tests/test_document_indexing.py`：上传 md → 调 `indexing_service` → 断言 `status=indexed`、chunks 生成、内容含原文、embedding 写入
3. `tests/test_document_replace.py`：上传 `员工手册.md` → 再上传同名 → 断言新 `version+1`、旧 `status=deleted`、旧 chunk 不命中检索、chunks 不重复
4. `tests/test_rag_refusal.py`：索引文档 → 问无依据问题 → 断言检索低于阈值 → 拒答文案；隐私问题 → `REFUSAL_PRIVACY`

**改动文件**：新建 `tests/conftest.py`、`tests/test_document_indexing.py`、`tests/test_document_replace.py`、`tests/test_rag_refusal.py`；可能 `pyproject.toml`（testcontainers）
**复用现有抽象**：`services/indexing_service` / `services/document_service` / `services/retrieval_service` / `services/chat_service`；集成测试可用现有 mock embedding 跑（真实语义另配 key）
**风险**：依赖本地 `docker compose up postgres`；测试须幂等；conftest 集中 fixture 注意体积红线
**验收**：`cd apps/api && uv run pytest` 全绿（含新增 3 个集成测试），且不破坏现有 4 个单测

### 3.4 增强项 4：前端引入 shadcn/ui

**目标**：统一组件体系，按开发规范引入 shadcn/ui（及可选 TanStack Table、React Hook Form + Zod）。

**现状**：无 `components.json`、无 `components/` 目录，依赖无 shadcn 生态；4 个 admin 页面纯 Tailwind 手写。

**实现步骤**
1. 初始化：`pnpm dlx shadcn@latest init`（生成 `components.json` / `lib/utils.ts` / CSS 变量，加依赖 `class-variance-authority` `clsx` `tailwind-merge` `lucide-react`）—— 注意 Tailwind v4 + Next 16 + React 19 兼容
2. 按需 add 组件：`pnpm dlx shadcn@latest add button card table input label badge dialog`
3. 逐页重构（保持功能不破）：
   - `app/admin/documents/page.tsx`（原生 table → shadcn Table；上传 → Dialog）
   - `app/admin/logs/page.tsx`（原生 table → shadcn Table + Badge 状态）
   - `app/admin/settings/page.tsx`（Card 容器）
   - `app/admin/chat/page.tsx`（Input / Button / Card）
4.（可选）TanStack Table（日志页排序分页）+ RHF + Zod（上传表单校验）

**改动文件**：新建 `components.json` / `lib/utils.ts` / `components/ui/*`；`package.json`（依赖）；`app/globals.css`（CSS 变量）；4 个 admin 页面重构
**复用现有抽象**：`lib/api.ts` 请求封装、现有页面业务逻辑
**风险**：Tailwind v4 + shadcn 配置坑（`AI_USAGE.md` 提到 pnpm/Tailwind 踩坑）；重构须功能不破；单文件 ≤ 500 行
**验收**：`pnpm build` + `pnpm lint` 通过；4 个 admin 页面手动走查功能正常

### 3.5 增强项 5：可观测性（Langfuse/OTel）+ Evals

**目标**：补 LLM 调用链路可观测 + 自动化评测，提升可维护性与质量度量。

**现状**：仅 loguru + trace_id，无任何 telemetry/exporter，前端无 telemetry。

**实现步骤（推荐先做后端 Langfuse）**

**A. Langfuse（推荐，对 LLM 友好）**
1. `uv add langfuse`
2. `core/config.py` 加可选 `LANGFUSE_PUBLIC_KEY` / `LANGFUSE_SECRET_KEY` / `LANGFUSE_HOST`（未配则 noop，不影响 mock 流程）
3. 在 `llm/openai_compatible.py`、`llm/embedding.py` 真实调用处包 Langfuse span（记录 prompt/completion/tokens/cost/latency），用现有 `trace_id` 关联
4. `services/chat_service.py` 顶层 span 串联 user → tool → answer

**B. 或 OTel（更通用）**
1. `uv add opentelemetry-distro opentelemetry-instrumentation-fastapi opentelemetry-exporter-otlp`
2. `opentelemetry-bootstrap -a install`
3. `app/main.py` 启动初始化 TracerProvider + FastAPI instrumentation + OTLP exporter
4. 在 `agent` / `chat_service` / `retrieval` 关键 span 加 `start_as_current_span`

**C. Evals 评测脚本**
1. 新建 `scripts/eval.py`，定义评测集（README 9 题 + 预期：命中 / 拒答 / 工具选对）
2. 真实 key 模式下自动评分，输出报表

**改动文件**：`pyproject.toml`、`core/config.py`、`llm/openai_compatible.py`、`llm/embedding.py`、`services/chat_service.py`、（OTel）`main.py`、新建 `scripts/eval.py`、`.env.example`
**复用现有抽象**：loguru `trace_id`（关联 span）、`chat_service` 主链路
**风险**：未配置时必须 noop（不破坏现有 mock 全流程）；Evals 依赖真实 key
**验收**：Langfuse/OTel 面板可见 trace；`python scripts/eval.py` 输出评分报表

## 五、推进建议

1. **分项迭代**：每项单独推进，**每项 1–3 个规范化 commit**
   - 示例：`feat(rag): add real token streaming`、`test(api): add rag refusal and replace integration tests`、`feat(web): adopt shadcn/ui for admin pages`、`feat(observability): add langfuse tracing`
2. **外部资源项**：需用户提供 key / 钉钉应用的项（1、2），由用户配置后再做真实验证
3. **完工自检**（提交前强制）：
   - `uv run ruff check . && uv run mypy app`（后端）
   - `uv run pytest`（后端，不破坏既有测试）
   - `pnpm lint` / `pnpm build`（前端，若动前端）
   - `git log -p | grep -iE "api_key|secret|sk-"`（无明文密钥）
   - 单文件 ≤ 500 行、单目录 ≤ 8 文件
4. **降级原则**：所有增强保持「未配置时降级为 mock」，不破坏当前无 key 全流程可跑的核心特性（这是项目最得意的技术决策，见 `自评.md`）
5. **推荐起步**：若希望立即产出代码，建议从 **项 3（集成测试）** 开始——纯后端、可独立、不依赖外部资源、直接提升质量信心。
