审查日期：2026-06-21  
审查范围：当前工作区代码、`招聘笔试题.html`、`开发规范.md`、核心后端/前端实现、配置、脚本、文档与部分 git 历史。

## 1. 总体结论

当前代码整体符合招聘笔试题的主要交付要求和 `开发规范.md` 的大部分强制约束：后端采用 FastAPI + Pydantic v2 + SQLAlchemy async + pgvector + Redis，前端采用 Next.js 16 + React 19 + Tailwind v4 + pnpm；已实现文档知识库、RAG 引用与拒答、Tool Calling Agent、钉钉/飞书 IM、Web 管理后台、日志、配置外置和自动化测试文件。

未发现以下一票否决项：`.env` 入库、真实 API Key 明文入库、主分支单一 init commit、无 IM 接入、工具选择在生产主链路硬编码、单文件超过 500 行、单目录超过 8 文件。

但存在 2 个需要修复的规范风险：

| 严重级别 | 问题 | 影响 |
|---|---|---|
| 高 | IM 验签缺失生产环境强制失败逻辑 | 若生产环境密钥仍为占位值，钉钉/飞书验签会直接放行，不符合“未配密钥仅开发环境放行，生产必须配齐” |
| 中 | 常规删除文档时未同步软删除 `document_chunks.deleted_at` | 检索不会命中旧文档，但不完全符合“文档/分块软删除”规范，审计一致性不足 |

另有 2 个验证环境风险：本机 `uv` 命令不可用，`apps/api/.venv` 的 Python 启动器指向不可用的 uv 托管 Python 路径，导致 pytest/mypy 在当前机器无法执行；前端构建通过，但 Next.js 提示 TypeScript 5.0.2 低于推荐 5.1.0。

## 4. 主要问题

### P1：生产环境 IM 密钥未配置时仍会跳过验签

位置：

- `apps/api/app/im/dingtalk.py:21` `verify_sign()`
- `apps/api/app/im/dingtalk.py:40` `verify_event_signature()`
- `apps/api/app/im/feishu.py:66` `verify_token()`
- `apps/api/app/im/feishu.py:80` `verify_signature()`

这些函数在密钥未配置时直接返回 `True`，只记录“仅开发环境允许”的 warning，但没有检查 `settings.is_dev`。如果 `APP_ENV=production` 且密钥仍为 `replace_me`，回调验签会被绕过。

建议：

```python
if not settings.is_secret_configured(secret):
    if settings.is_dev:
        logger.warning(...)
        return True
    return False
```

或抛出 `AppException(ErrorCode.IM_VERIFY_ERROR, ...)`，并为生产分支补充单测。

### P2：删除文档未同步标记对应 chunks 的 `deleted_at`

位置：

- `apps/api/app/services/document_service.py:95` `soft_delete_document()`

常规删除只设置了 `doc.status = "deleted"` 和 `doc.deleted_at`。当前检索 SQL 通过 join documents 并过滤 `d.deleted_at IS NULL`，所以旧 chunk 不会参与检索；但 `开发规范.md` 要求 chunk 层也有软删除标记，且项目文档强调 `document_chunks.deleted_at` 支撑分块级软删除。

建议在 `soft_delete_document()` 中同步执行：

```python
await session.execute(
    update(DocumentChunk)
    .where(DocumentChunk.document_id == doc.id, DocumentChunk.deleted_at.is_(None))
    .values(deleted_at=datetime.now(UTC))
)
```

并补一个“删除文档后 chunks.deleted_at 被置值”的测试。

## 7. 建议修复顺序

1. 先修 IM 生产验签强制失败逻辑，并加生产环境单测。
2. 再修普通删除文档时同步软删除 chunks，并补数据一致性测试。
3. 修复本地 uv 环境：安装 `uv` 或重建 `apps/api/.venv`，保证 `uv run pytest && uv run mypy app` 可复现。
4. 升级/锁定 TypeScript 至 Next 推荐版本，例如 `typescript>=5.1`。
5. 在真实 LLM/Embedding 与钉钉/飞书沙箱中跑一遍题目 10 个验收问题，并把结果记录到文档或 eval 报告。

## 8. 审查结论

从静态代码、配置和前端构建结果看，项目已达到笔试题核心要求，且具备不少加分项。当前最需要处理的是安全边界和数据软删除一致性两处问题；修复后，再补齐当前机器上的 uv 测试可复现性，整体规范符合度会更稳。
